# OMEN – Auto-refreshing China event timeline (Grok Live Search)

**Date:** 2026-07-18
**Files:** `omen/update-china-events.py` (new), `omen/china-ai-monitor.html`, `.github/workflows/refresh.yml`, `omen/china-events.json` (new, seeded)
**Live site:** https://omen-ai.blankm.workers.dev/china-ai-monitor

---

## The problem

The "Chinese model catalysts vs US stock reactions" timeline was a hand-curated,
hardcoded `TIMELINE` array whose last entry was 2026-07-07. It never refreshed — so it
missed **Kimi K3** (launched 2026-07-16), exactly the kind of catalyst the page exists to
track.

## The design: hybrid, not full auto

A fully machine-generated timeline is the wrong call for a public, finance-adjacent page —
the curated entries carry precise stock-move claims ("NVDA −16.9%") that must stay
trustworthy, and an LLM will occasionally invent numbers. So:

- **Curated base stays authoritative.** The `TIMELINE` const in `china-ai-monitor.html` is
  unchanged — it remains the trusted history *and* the fallback.
- **Recent half is auto-generated** into `china-events.json` and merged on top. This uses
  the same override pattern the page already uses for `china-data.json`.

## How it refreshes

`omen/update-china-events.py` (new) calls the **Grok Live Search** API — the same
`XAI_API_KEY` already used by `update-influencers.py` — to fetch recent Chinese-model
catalysts (last ~45 days) as `{d, t, rx, src}` events, and writes `china-events.json`.

The CI cron (`refresh.yml`) already runs every 30 min in market hours / hourly overnight
and commits data back (which triggers the Cloudflare rebuild). A new step runs the events
updater, gated on `XAI_API_KEY`; `china-events.json` was added to the commit list. The
script **self-gates to run the paid model call at most once per ~24h** (`MIN_AGE_HOURS=20`,
checked against the file's `generated_at`), so "refresh every 24 hours" holds even though
the cron fires far more often. `--force` overrides.

`worker.js` needs no change: like `china-data.json`, the events file is served from the
bundled assets (not R2), refreshed via the commit→push→rebuild path.

## Guardrails (this is a public page — no invented numbers, no injection)

- **No fabricated figures.** The prompt forbids stating a stock-move % unless actually
  reported; otherwise a qualitative note ("no clear indexed move"). Every event carries a
  **source URL** from live search so a reader can verify.
- **Validation.** Events are dropped unless the date is ISO, strictly after the curated
  cutoff (`SINCE=2026-07-07`), and not future-dated; headlines must be non-trivial; source
  URLs must be `http(s)` or are blanked.
- **XSS-safe render.** Curated entries keep their trusted `<b class='up/dn'>` markup;
  auto entries are **fully HTML-escaped** on render (`esc()` on every field), and only an
  `http(s)` source link is allowed. Verified: an injected `<img onerror>` / `<script>` /
  `javascript:` src renders as inert text and does not execute.
- **Fail-safe.** No `XAI_API_KEY`, an API error, or nothing valid → the existing file is
  left untouched and the page falls back to the curated timeline. It never regresses.
- **No data loss.** New events are *unioned* with the previous file (deduped by date +
  headline, retained ~150 days, capped 12), so a catalyst added last week doesn't vanish
  once it ages out of the search window.

## What the reader sees

Recent auto entries append below the curated ones (timeline is oldest→newest), each with a
small amber **"live"** tag and a **source ↗** link, plus a note under the header:
*"N recent entries auto-updated daily via Grok Live Search (asof …) · older entries
curated. live entries are machine-added – verify via source."*

## Bootstrap

`china-events.json` is seeded by hand with the real, sourced **Kimi K3** entry so it shows
immediately, with an intentionally old `generated_at` so the **next CI run regenerates it
via Grok** and takes over (provenance flips from "manual seed" to "x.ai Live Search"). If
Grok's window later misses K3, the union logic keeps the seeded entry.

## Keeping the cutoff in sync

`SINCE` in `update-china-events.py` (and the implicit "last curated date" the page computes)
must track the newest curated `TIMELINE` entry. If you ever extend the curated base past
2026-07-07, bump `SINCE` so the two halves don't overlap.

## Verification

- Script: `--sample` writes a valid file; no-key / API-error / empty → no-op; validation
  drops pre-cutoff / malformed / future events and sanitizes bad URLs; 24h gate works.
- Page (local, DOM-verified): 8 events render (7 curated + 1 auto), correct order, "live"
  tag + source link present, curated bold up/down markup intact, freshness note correct.
- Security: injected HTML / script / `javascript:` URL in an auto event is escaped and
  does not execute.
